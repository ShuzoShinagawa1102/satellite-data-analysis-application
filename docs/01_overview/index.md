# 01_overview / index

## 文書名
危険・違法盛土 監視運用ダッシュボード（盛土監視Ops）  
**完全仕様書（React + Django）**

## 本書の位置づけ
本書は、自治体向けの「衛星データ×AIを活用した盛土監視運用ダッシュボード」について、**ビジネス課題（Why）・解決仕様（What）・アーキテクチャ（How）** を統合した仕様書である。

- 本書は **提案書ではない**
- 本書は **開発見積・工数・スケジュールの提示を目的としない**
- 本書は **プロダクトの仕様確定・認識合わせ** を目的とする

## 対象プロダクト（正式名称・仮）
**盛土監視Ops（Morido Watch Ops）**

## プロダクトの定義（1文）
衛星データおよびAIによる疑義地点検知を起点に、自治体の監視・トリアージ・現地確認・案件管理・証跡化までを一体化する、**盛土監視業務専用の運用ダッシュボード**。

---

## 1. 背景（要約）
盛土規制法の運用進展に伴い、自治体では「危険・違法な盛土等の早期発見」「監視」「現地確認」「記録」「継続管理」の実務負荷が増加している。  
一方で、広域を人手のみで日常監視することには限界があり、衛星データ活用の有効性が高まっている。

本プロダクトは、**衛星データ解析単体ではなく、自治体の監視運用業務そのもの**を対象に設計する。

## 4. 設計原則（本プロダクトの思想）
本プロダクトの設計判断は、以下の原則に従う。

### 4.1 課題ドリブン
「衛星が使えるから作る」のではなく、**自治体監視業務のボトルネック**を解消するために作る。

### 4.2 Human-in-the-loop
AIは「疑義地点の抽出・優先順位付け」を担い、最終判断は人が行う。  
本プロダクトは **人の判断を強く支援するUI/証跡** を中核価値とする。

### 4.3 運用OS志向
検知画面のみで終わらず、以下を一気通貫で扱う。

- 検知
- トリアージ
- 現地確認
- 案件化
- 継続監視
- 証跡/監査

### 4.4 説明責任を前提とした設計
行政実務では、後から「なぜその判断をしたか」が問われる。  
よって、**判断根拠・変更履歴・担当者記録** を省略しない。

### 4.5 段階的高度化
初期仕様では、過剰な自動化よりも **誤検知に強い運用設計** を優先する。  
高度な最適化（再学習UI、複雑な自動ワークフロー）は後続拡張とする。

## 5. 想定ユーザーと利用シーン（概要）
### 主ユーザー
- 監視担当（主担当）
- 現地確認担当
- 管理職（統括）
- 許認可照合担当/関係部署
- 監査・レビュー観点の閲覧者（必要に応じて）

### 代表利用シーン
1. AI候補地点の毎日/毎週確認
2. 現地確認対象の優先順位付け
3. 現地確認結果の登録
4. 案件としての継続管理
5. 月次・四半期の業務レポート作成
6. 監査・説明用の証跡参照

## 7. 重要な確定事項（仕様判断）
本書の前提として、以下は確定とする。

- **技術スタックは React + Django 固定**
- プロダクトは **自治体向けWebダッシュボード** を中心に設計する
- AIは **最終判定を代替しない**
- プロダクト価値の中心は **運用業務の統合** に置く
- 初期仕様は **盛土監視領域に特化** する（横展開は考慮のみ）

---

## 8. 成果物としての期待
本書の完成状態は以下を満たすこと。

- 問題領域が曖昧でない（誰のどの業務課題を解くかが明確）
- 画面/機能/データ/状態遷移が一貫している
- React + Django で実装可能な粒度に落ちている
- 非機能要件（監査・セキュリティ・運用）が抜けていない
- 外部説明・内部設計の双方で使える

---

## 9. 用語の簡易定義（本書内）
- **盛土等**: 盛土・切土・土石の堆積等を含む、本領域で監視対象となる行為・状態の総称
- **疑義地点**: AIやルールで「確認対象」として抽出された地点（違法確定ではない）
- **トリアージ**: 複数候補の優先順位付け
- **案件**: 調査・確認・対応の管理単位
- **現地確認**: 現地巡回・写真取得・メモ記録・一次判断の業務
- **証跡**: 判断根拠や履歴として後から参照可能な記録（画像・ログ・メモ等）
